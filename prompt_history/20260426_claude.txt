3. 配布用バイナリ化（.exe化）の方法
PowerShellスクリプトをそのまま渡すと、実行ポリシー（Execution Policy）に阻まれて動かないことが多いため、以下のいずれかでラップして配布するのが一般的です。

A. ps2exe (推奨)
PowerShellスクリプトをそのまま.exeに変換するツールです。

利点: アイコンを設定でき、右クリック「PowerShellで実行」の手間が省けます。

方法: Install-Module ps2exe 実行後、ps2exe ./myscript.ps1 ./setup.exe で生成。

B. IExpress (Windows標準)
Windowsに標準搭載されている「自己解凍インストーラー作成ツール」です。

Win + R -> iexpress で起動。

スクリプトを同梱し、実行コマンドとして powershell -ExecutionPolicy Bypass -File setup.ps1 を指定します。

4. 運用のヒント
WSL機能自体の有効化: ユーザーのPCでWSLが有効になっていない場合を想定し、スクリプトの冒頭で wsl --status を確認し、エラーなら「Windowsの機能の有効化」を促すメッセージを出すと親切です。

ポートの競合: localhost:3001 が既に使われている場合、npm run dev が失敗します。スクリプト側で Test-NetConnection -Port 3001 を使ってチェックするガードコードを入れるのが定石です。

1. 実行ポリシー（Execution Policy）の回避

ps1の場合: 青いスクリプトアイコンのままです。

## 全体方針：賛成だが「.exe単体」より「.exe + 常駐UI」を推奨

PowerShell + ps2exe の方向性は良いですが、`Read-Host` で待たせる構成は配布物としては弱いです。**WPF or Windows Forms を埋め込んだPowerShell** か、**タスクトレイ常駐**にすると、停止操作が直感的になります。

### 推奨構成

| 役割 | 技術 | 理由 |
|---|---|---|
| エントリ | `.exe`（ps2exe で生成） | SmartScreenは出るが体験は最良 |
| GUI | PowerShell + WPF (XAML) | 「起動中」「停止」ボタンを表示。Read-Hostより遥かにマシ |
| WSL制御 | `wsl --import` / `-d` / `--terminate` | Geminiの案通り |
| 配布 | R2 から `.tar` をDL | OK。ただし**再開可能DL**と**SHA256検証**必須 |

## Geminiの案で見落とされている重要点

### 1. tar の整合性検証（必須）
3GBのDLは途中で壊れる可能性大。
```powershell
$expected = "abc123..."
$actual = (Get-FileHash $tarFile -Algorithm SHA256).Hash
if ($actual -ne $expected) { throw "破損" }
```

### 2. 再開可能ダウンロード
`Invoke-WebRequest` は3GBには不向き（メモリ食う・進捗遅い）。**`BITS` (`Start-BitsTransfer`)** を推奨：
```powershell
Start-BitsTransfer -Source $r2Url -Destination $tarFile -Description "イメージDL中"
```
進捗ダイアログも自動で出ます。

### 3. WSL未インストール環境への対応
`wsl --status` だけでなく、**WSL2カーネル**が必要です。Win10/11問わず、未導入PCでは：
```powershell
wsl --install --no-distribution
```
で済むケースが多い（要再起動）。これを案内するロジックを入れる。

### 4. ディストリ名の衝突回避
`MyMediaSystem` のような固定名だと、再インストール時に `--unregister` が必要。**バージョン付きの名前**（例：`homepage-v2-2026`）にし、アップグレード時の挙動を設計しておく。

### 5. `npm run dev` をバックグラウンド起動する罠
`Start-Process wsl ... -WindowStyle Hidden` ではWSL内のNode.jsプロセスが**親PowerShellと一緒に死なない**ことがあります。停止時は：
- `wsl --terminate <distro>` で**確実に**全プロセスkill
- ポート3001の解放も確認

### 6. 二重起動防止
**Mutex** で .exe 自体の二重起動も防ぐ：
```powershell
$mutex = New-Object System.Threading.Mutex($false, "Global\HomepageV2Installer")
if (-not $mutex.WaitOne(0)) { exit }
```

### 7. ログ出力
ユーザーから「動かない」と言われた時のため、`%LOCALAPPDATA%\homepage-v2\install.log` に必ず出力。

## ps2exe vs IExpress：私もAを推す（理由は別）

Geminiの理由（アイコン・ポリシー回避）に加えて：

- **IExpressは古く、Defenderの誤検知率が高い**（自己解凍SFXは怪しまれやすい）
- **ps2exe生成のexeも誤検知されることがある**ので、配布前に [VirusTotal](https://www.virustotal.com/) で確認すべき
- 誤検知が酷い場合は **Inno Setup** に切り替える選択肢もある（OSSで超メジャー、Defender実績豊富、UIも作れる）

## 別案：Inno Setup を強く推奨したい

正直、3GBのDL・整合性検証・WSLインポート・アンインストーラ生成…まで考えると、**Inno Setup + 内部からPowerShell呼び出し**が最も無難です。

| 項目 | ps2exe単独 | Inno Setup |
|---|---|---|
| アンインストーラ自動生成 | ✗（自作） | ✓ |
| スタートメニュー登録 | ✗（自作） | ✓ |
| 進捗UI | 自作 | 標準提供 |
| Defender誤検知 | 中 | 低（実績多い） |
| 学習コスト | 低 | 中 |

「セットアップ画面の起動・停止」用には別途、軽量な**WPF常駐アプリ（ps2exe製）**をスタートメニューから起動、という二段構成が綺麗です。
