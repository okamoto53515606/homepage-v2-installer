3. 配布用バイナリ化（.exe化）の方法
PowerShellスクリプトをそのまま渡すと、実行ポリシー（Execution Policy）に阻まれて動かないことが多いため、以下のいずれかでラップして配布するのが一般的です。

A. ps2exe (推奨)
PowerShellスクリプトをそのまま.exeに変換するツールです。

利点: アイコンを設定でき、右クリック「PowerShellで実行」の手間が省けます。

方法: Install-Module ps2exe 実行後、ps2exe ./myscript.ps1 ./setup.exe で生成。

B. IExpress (Windows標準)
Windowsに標準搭載されている「自己解凍インストーラー作成ツール」です。

Win + R -> iexpress で起動。

スクリプトを同梱し、実行コマンドとして powershell -ExecutionPolicy Bypass -File setup.ps1 を指定します。

4. 運用のヒント
WSL機能自体の有効化: ユーザーのPCでWSLが有効になっていない場合を想定し、スクリプトの冒頭で wsl --status を確認し、エラーなら「Windowsの機能の有効化」を促すメッセージを出すと親切です。

ポートの競合: localhost:3001 が既に使われている場合、npm run dev が失敗します。スクリプト側で Test-NetConnection -Port 3001 を使ってチェックするガードコードを入れるのが定石です。

プロセスの二重起動防止: スクリプト開始時に wsl --list --running を確認し、既に動いていればインポートと起動をスキップしてブラウザだけ開くようにすると、より洗練された挙動になります。

1. 実行ポリシー（Execution Policy）の回避
Windowsの標準設定では、ネットから拾ってきた .ps1 ファイルはセキュリティ制限で実行できません。

2. アイコンやプロパティのカスタマイズ
自作の個人メディアシステムであれば、ブランディングは重要ですよね。

exeの場合: 独自のアイコン画像（.ico）を設定できます。

ps1の場合: 青いスクリプトアイコンのままです。

「怪しい警告」が出たとしても、アプリのアイコンがしっかり設定されているだけで、ユーザーの心理的な安心感（＝信頼性）は大きく変わります。

exeの場合: スクリプトがカプセル化されるため、うっかり書き換えられたり、R2のURLやトークンなどが不用意に露出するのを（気休め程度ではありますが）防げます。

.exe 自体に「ウィンドウを表示しない（WindowStyle Hidden）」設定を組み込めるため、バックグラウンドでWSLを動かしつつ、ユーザーにはブラウザだけを見せる、といったスマートな挙動が作りやすいです。

## 全体方針：賛成だが「.exe単体」より「.exe + 常駐UI」を推奨

PowerShell + ps2exe の方向性は良いですが、`Read-Host` で待たせる構成は配布物としては弱いです。**WPF or Windows Forms を埋め込んだPowerShell** か、**タスクトレイ常駐**にすると、停止操作が直感的になります。

### 推奨構成

| 役割 | 技術 | 理由 |
|---|---|---|
| エントリ | `.exe`（ps2exe で生成） | SmartScreenは出るが体験は最良 |
| GUI | PowerShell + WPF (XAML) | 「起動中」「停止」ボタンを表示。Read-Hostより遥かにマシ |
| WSL制御 | `wsl --import` / `-d` / `--terminate` | Geminiの案通り |
| 配布 | R2 から `.tar` をDL | OK。ただし**再開可能DL**と**SHA256検証**必須 |

## Geminiの案で見落とされている重要点

### 1. tar の整合性検証（必須）
3GBのDLは途中で壊れる可能性大。
```powershell
$expected = "abc123..."
$actual = (Get-FileHash $tarFile -Algorithm SHA256).Hash
if ($actual -ne $expected) { throw "破損" }
```

### 2. 再開可能ダウンロード
`Invoke-WebRequest` は3GBには不向き（メモリ食う・進捗遅い）。**`BITS` (`Start-BitsTransfer`)** を推奨：
```powershell
Start-BitsTransfer -Source $r2Url -Destination $tarFile -Description "イメージDL中"
```
進捗ダイアログも自動で出ます。

### 3. WSL未インストール環境への対応
`wsl --status` だけでなく、**WSL2カーネル**が必要です。Win10/11問わず、未導入PCでは：
```powershell
wsl --install --no-distribution
```
で済むケースが多い（要再起動）。これを案内するロジックを入れる。

### 4. ディストリ名の衝突回避
`MyMediaSystem` のような固定名だと、再インストール時に `--unregister` が必要。**バージョン付きの名前**（例：`homepage-v2-2026`）にし、アップグレード時の挙動を設計しておく。

### 5. `npm run dev` をバックグラウンド起動する罠
`Start-Process wsl ... -WindowStyle Hidden` ではWSL内のNode.jsプロセスが**親PowerShellと一緒に死なない**ことがあります。停止時は：
- `wsl --terminate <distro>` で**確実に**全プロセスkill
- ポート3001の解放も確認

### 6. 二重起動防止
**Mutex** で .exe 自体の二重起動も防ぐ：
```powershell
$mutex = New-Object System.Threading.Mutex($false, "Global\HomepageV2Installer")
if (-not $mutex.WaitOne(0)) { exit }
```

### 7. ログ出力
ユーザーから「動かない」と言われた時のため、`%LOCALAPPDATA%\homepage-v2\install.log` に必ず出力。

## ps2exe vs IExpress：私もAを推す（理由は別）

Geminiの理由（アイコン・ポリシー回避）に加えて：

- **IExpressは古く、Defenderの誤検知率が高い**（自己解凍SFXは怪しまれやすい）
- **ps2exe生成のexeも誤検知されることがある**ので、配布前に [VirusTotal](https://www.virustotal.com/) で確認すべき
- 誤検知が酷い場合は **Inno Setup** に切り替える選択肢もある（OSSで超メジャー、Defender実績豊富、UIも作れる）

## 別案：Inno Setup を強く推奨したい

正直、3GBのDL・整合性検証・WSLインポート・アンインストーラ生成…まで考えると、**Inno Setup + 内部からPowerShell呼び出し**が最も無難です。

| 項目 | ps2exe単独 | Inno Setup |
|---|---|---|
| アンインストーラ自動生成 | ✗（自作） | ✓ |
| スタートメニュー登録 | ✗（自作） | ✓ |
| 進捗UI | 自作 | 標準提供 |
| Defender誤検知 | 中 | 低（実績多い） |
| 学習コスト | 低 | 中 |

「セットアップ画面の起動・停止」用には別途、軽量な**WPF常駐アプリ（ps2exe製）**をスタートメニューから起動、という二段構成が綺麗です。

## 切り分けの見立て（可能性が高い順）

1. **DL実装が一番怪しい**  
   Inno Setupの `DownloadPage`（内部は WinHTTP のシングルストリーム）を使っています。
   - 単一TCPコネクション・固定バッファで、**並列/レンジ分割なし**。Cloudflare R2 は1コネクションあたり数MB/s程度に絞られることがあり、回線が太くても伸びません。
   - tar（おそらく数百MB～GB級）だと体感で「異常に遅い」になりがちです。
   - 対策候補: 
     - `curl.exe`（Windows 10 1803+ 標準）で `--http2 -C -`（レジューム）
     - `aria2c` を同梱して `-x 8 -s 8 -k 1M`（並列8コネクション、これが一番効く）
     - 最低でも `Start-BitsTransfer`（複数セッション、レジューム可、低優先度回避）
   - 経験上、R2 pub URL に対して aria2c の `-x 8` で **5～10倍** 速くなるケースが多いです。

2. **R2側の要因**  
   `pub-xxx.r2.dev` は **開発/小規模配布向け** の無料公開URLで、Cloudflareがレート制限・帯域シェーピングを行う想定です。本番配布用にはカスタムドメイン＋CDN（Cache Rules で `Cache-Control: public, max-age=...` を効かせる）に載せ替える方が安定します。
   - 1回目はオリジン取得で遅くても、2回目以降キャッシュHITで高速化、という挙動も pub URL では弱い。
   - HEAD で `cf-cache-status` と `Accept-Ranges: bytes` を確認するのが先決。Range が効くなら並列DLで一気に改善します。

3. **Windows Defender / SmartScreen**  
   - Defender のリアルタイムスキャンは **書き込み中のtarを逐次スキャン**するため、巨大tarだと**書き込みスループットが落ちます**（CPU 1コアが `MsMpEng.exe` で張り付く）。タスクマネージャで `Antimalware Service Executable` が高ければクロ。
   - SmartScreen は exe/MSI 主体でtar自体には基本効きませんが、署名なしインストーラ全体の起動が遅れることはあります。
   - 対策: インストーラのキャッシュフォルダ（例: `%ProgramData%\HomepageV2\cache`）を**Defender除外**に追加、または**署名済みtar.gzに圧縮**して転送量を半減（tarならgzipで30-60%縮むことが多い）。

## 優先順位の提案

1. **まず計測**: `curl.exe -o NUL https://pub-...tar` で素の速度、`curl --range 0-10485760` でRange可否、HEAD で `cf-cache-status` を確認。

GitHub Copilot: ## なぜ `wsl --install` は速くてカスタムtarのDLは遅いのか

`wsl --install`（既定の Ubuntu）は、Microsoft が **Microsoft Store / Azure Front Door (AFD) CDN** から配信しています。これは:

- 世界中にエッジが分散した本物のCDN（日本ならまず東京/大阪エッジから配信）
- HTTP/2、巨大なエッジキャッシュ、十分な帯域シェーピング
- Windows Update の配信基盤と同等で、家庭回線でも数十～数百Mbps出る

一方、`pub-xxx.r2.dev` は:

- Cloudflare R2 の **開発者向け公開URL**で、本番配布用ではない（Cloudflare自身が "for development purposes" と明記）
- レート制限あり（公開ドキュメント上も "rate limits apply"）
- カスタムドメイン+Cache Rulesを通さないと、CDNキャッシュが十分効かない構成になりがち
- 1コネクションあたりの帯域が絞られやすい

そこに **Inno Setup の DownloadPage（シングルストリーム・レンジなし）** が重なるので、`wsl --install` の体感（CDN+OS最適化されたDL）と比べて極端に遅く見える、というのが構造的な答えです。**Defender のリアルタイムスキャンも追い打ち**になりますが、本質はネットワーク側です。

## 今のDLは中止すべきか？

**中止していいです。** むしろ計測のために中止推奨。理由:

1. 今のDLは Inno Setup 内蔵DL = 計測したい「素のR2の速さ」ではなく「Inno DL実装込みの速さ」しか測れない。切り分けにならない。
2. R2 pub URL は **Range リクエスト対応**なので、`curl.exe -C -` で**続きから再開DL可**。これまでのDLが完全に無駄になるわけではありません（インストーラのキャッシュに残っていれば、後でリネームして使い回せる可能性もあり）。
3. tar.sha256 は数十バイトなので即DLできます。

## 結論

**R2 pub URL の帯域シェーピングが主犯**です。DL実装の問題（Inno Setupのシングルストリーム）も二次的にありますが、本質はR2側。理由:

## SHA256 ファイルは必須か？

**必須ではないが、強く推奨**です。理由:

- **DL破損検知**: 大きいファイルなのでネットワーク途中切断・キャッシュ破損が起こり得る。`wsl --import` で破損tarを食わせると分かりにくいエラーが出ます。
- **改ざん検知**: HTTPSがあるので中間者攻撃は基本防げますが、配信元の事故（誤ったファイルを上げた等）も検知できます。
- **インストーラの既存実装**: 現状の homepage-v2-installer.iss がすでに `.sha256` を前提にDL・検証している作りなので、廃止するとコード変更箇所が増える。**残すのが楽**。

GitHub Releases自体はファイル単位のチェックサムを公開していないので、自前で `.sha256` を併置するのが定石です。

## SHA256 の作り方

### WSL/Linux 側（今いる場所）

```bash
cd /mnt/d/wsl_backup

# gz に対してハッシュを取る（配布するのは .gz のほうなので）

# 中身確認
cat homepage-v2-latest.tar.gz.sha256
# 例: 3a7bd9...e2  homepage-v2-latest.tar.gz
```

### PowerShell 側（Windowsで作る場合）

```powershell
$f = 'D:\wsl_backup\homepage-v2-latest.tar.gz'
$hash = (Get-FileHash $f -Algorithm SHA256).Hash.ToLower()
"$hash  $(Split-Path $f -Leaf)" | Set-Content -NoNewline "$f.sha256" -Encoding ASCII
```

## ファイル形式の注意点

既存の `homepage-v2-latest.tar.sha256`（64バイト）は**おそらく `.tar` に対するハッシュ**です。**配布物を `.tar.gz` に変更するなら、ハッシュも .gz に対して取り直す必要があります**。混在すると検証失敗します。

## sha256 ファイルの中身フォーマット

`sha256sum` 標準形式は:

```
<64桁hex>  <ファイル名>
```

（hexと名前の間はスペース2つ）。インストーラ側の検証コードがこの形式をパースしているか、それとも先頭64桁だけ見ているかで、形式の自由度が変わります。一応見ておきましょう:

`LoadSha256FromFile` は **「先頭からスペースまでをハッシュとして取り出して小文字化」** する実装なので、`sha256sum` の標準出力フォーマット（`<hex>  <filename>`）でそのままOKです。

## バージョンは分けたほうがいいか？ → **強く推奨します**

理由:

| 観点 | 内容 |
|---|---|
| **更新頻度の違い** | WSLイメージ（中身のアプリ）は頻繁に更新したい。インストーラ本体（exe生成物・トレイアプリ・スクリプト）は構造変更時しか更新したくない。 |
| **配布物の重さ** | exeインストーラ自体は軽量（数MB）。WSLイメージは1.1GB。これらを同じバージョン番号で縛ると「軽微な中身更新でもインストーラ再配布」「インストーラのバグ修正のために中身も巻き込む」など噛み合わない。 |
| **ユーザー体験** | 既にインストーラを入れたユーザーは、起動時にWSLイメージだけ最新化できると嬉しい（将来の自動アップデート機能の余地）。 |
| **互換性管理** | インストーラ側がイメージのフォーマット要求（gzip対応バージョン等）を持つので、メジャー更新時のみインストーラ側も上げる、という運用が自然。 |

## 推奨される命名

| 種別 | 例 | 用途 |
|---|---|---|
| **インストーラのバージョン** | `installer-1.0.0` | exeの構造変更時のみインクリメント。Inno Setup の `MyAppVersion`。 |
| **WSLイメージのバージョン** | `image-2.0.0`, `image-2.0.1`, ... | 中身更新時に頻繁にインクリメント。 |
| **`latest` リリース** | 必ず最新 image を pre-release ではなく通常リリースとして公開すれば、`/releases/latest/download/...` で取得可能 |

ただし**注意点**として、GitHub Releases の `latest` は **リポジトリ単位で1つ**です。インストーラとイメージを別々に「最新」管理したい場合、選択肢:

1. **リポジトリを分ける**: `homepage-v2-installer`（exe配布用、めったに更新なし）と `homepage-v2-image`（WSL配布用、頻繁更新）。一番きれいだが管理対象が増える。
2. **同じリポジトリでタグ命名規則を分ける + 取得方法を変える**: たとえばインストーラexeはタグ `installer-vX.Y.Z`、イメージはタグ `image-vX.Y.Z` でリリース。`releases/latest` はイメージ用に使い、インストーラexeは固定タグでリンクする運用。**`/releases/latest` はあくまで「最新の通常リリース」を1つだけ指すので、片方しか latest 化できません**。
3. **API経由で動的解決**: インストーラ実行時に `https://api.github.com/repos/.../releases` を叩いてフィルタする。柔軟だが認証レート制限の注意要。

そしてURL組み立て箇所をパス形式の違いに対応:

```pascal

  ShaUrl := 'https://github.com/{#MyReleaseRepo}/releases/latest/download/{#MyTarFileName}.sha256';
end
else
begin
  TarUrl := 'https://github.com/{#MyReleaseRepo}/releases/download/{#MyReleaseTag}/{#MyTarFileName}';
  ShaUrl := 'https://github.com/{#MyReleaseRepo}/releases/download/{#MyReleaseTag}/{#MyTarFileName}.sha256';
end;
```

ただし `latest` 運用には **隠れた注意点**があります:

### latest運用の落とし穴

1. **インストーラexe本体に `MyAppVersion` が焼き込まれる**ので、中身だけ更新してもインストーラexeは再ビルド・再配布しないとユーザーには届かない（ユーザーが既にインストーラexeを保管していれば、再実行すると新しいイメージを取りにいく挙動になります）。
2. **SHA256検証は配布側で更新済みである必要がある**: 新リリースを公開するときは必ず `.tar.gz.sha256` も同じリリースに同梱すること。片方だけ更新するとインストーラが壊れる。
3. **古いインストーラexeで新しいイメージが取れてしまう**: 将来イメージ形式に破壊的変更（例: zstd採用）を入れた時、古いインストーラexeがそれを処理できずに失敗する。
   - 対策: イメージ側に「最低必要なインストーラバージョン」を入れたバージョンファイル（manifest.json的なもの）を別アセットで配り、インストーラ起動時に確認する。今は不要だが将来の拡張余地として認識しておけばOK。

## ② .exe をそのままアップ → **大丈夫、ただし警告は出ます**

GitHub Releases に `.exe` をアップロードすること自体は完全にOK（多くのOSSプロジェクトが同じやり方）。ただし**警告問題は配布形式とは別の話**です:

### 警告が出る本当の理由
**コード署名（Authenticode署名）がないから**です。zip にしても**警告は消えません**。むしろ:

| 配布形式 | DL時の挙動 | 実行時の挙動 |
|---|---|---|
| **`.exe` 裸** | ブラウザによっては「危険なファイル」警告が出る場合あり（Chrome/Edge） | SmartScreen の **「WindowsによってPCが保護されました」**画面（"Run anyway"が必要） |
| **`.zip` に入れる** | DL時の警告は出にくい（中身を判定しないブラウザが多い） | 解凍後の.exe実行時は **同じSmartScreen警告**が出る（mark-of-the-web引き継ぎ） |
| **コード署名付き`.exe`** | 警告ほぼなし | 署名検証OK→警告なし（EV証明書なら即時、OV証明書はreputation蓄積後に消える） |

つまり**zip化はDLの一手間を減らすかもしれないが、SmartScreen警告は消えない**。むしろzip→解凍の手順がユーザーに面倒なだけです。

### 現実的な判断

**`.exe` のまま配布で問題なし**、というのが私の推奨です。理由:

1. SmartScreen警告は署名がない限り避けられないので、形式変更で頑張っても効果薄。
2. ユーザーは「詳細情報 → 実行」でクリアできる。READMEに案内を書けば十分。
3. zip化すると、ユーザーに「右クリック → プロパティ → ブロック解除」を案内する必要が出るケースもあり、むしろ手数が増える可能性。
4. 多くの著名OSS（Inno Setup自体、7-Zip、Notepad++など）も裸の `.exe` で配布しています。

### 警告を本当に消したい場合の選択肢

- **コード署名証明書を購入**: OV証明書（年$100~200）かEV証明書（年$300~600）。SmartScreenを即時クリアしたいならEV。個人配布だと過剰投資なことが多い。
- **Microsoft Store**: 審査必要だが署名・配布が無料。ただし WSL のtar取り込みのような特権操作はストアアプリのサンドボックスから難しい。
- **winget manifest 登録**: コミュニティから信頼を獲得すればSmartScreenが緩和されることがある。ただし即効性なし。

User: WSL を使用して Windows に Linux をインストールする方法
開発者は、Windows マシン上で Windows と Linux の両方の機能に同時にアクセスできます。 Windows Subsystem for Linux (WSL) を使用すると、開発者は Linux ディストリビューション (Ubuntu、OpenSUSE、Kali、Debian、Arch Linux など) をインストールし、Linux アプリケーション、ユーティリティ、Bash コマンド ライン ツールを、従来の仮想マシンやデュアルブートのセットアップのオーバーヘッドなしで、変更なしで Windows に直接使用できます。

## 原因
